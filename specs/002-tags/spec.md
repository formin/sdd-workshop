# Feature Specification: ToDo 항목 태그 부여 및 태그 기반 필터

**Feature Branch**: `002-tags`
**Created**: 2026-05-03
**Status**: Draft
**Issues**: #1, #2
**Input**: User description: "기존 Todo 앱에 태그 기능을 추가하고 싶어. 추가할 기능: 1) Todo 항목 생성 시 태그를 선택적으로 추가할 수 있다 — 태그는 최대 5개까지 허용, 각 태그는 20자 이내, 태그 없이도 Todo 생성 가능 (기존 기능 유지). 2) 목록 조회 시 태그로 필터링할 수 있다 — `todo list --tag 태그명` 형식. 기존 Todo 기능(추가/조회/완료/삭제)은 그대로 유지되어야 함."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 항목 생성 시 태그 부여 (Priority: P1)

사용자는 새 ToDo 항목을 생성할 때 항목을 분류·식별하기 위한 짧은 라벨(태그)을 선택적으로 0~5개까지 부여할 수 있다. 태그를 지정하지 않으면 기존 흐름과 동일하게 항목이 생성된다.

**Why this priority**: 태그를 부여할 수 없으면 태그 기반 필터링(US2)도 무의미하므로 본 스토리가 모든 후속 가치의 전제이다. 동시에 기존 사용자의 "태그 없이 빠르게 추가" 흐름이 깨지지 않아야 하므로 P1.

**Independent Test**: 태그를 지정해 항목을 추가한 뒤 목록 조회 결과에서 해당 태그 정보가 보존·표시되는지, 그리고 태그 없이 추가한 항목도 정상 생성되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 빈 목록, **When** `todo add "보고서" --tag work --tag 2026Q2`, **Then** 새 항목이 두 개의 태그(`work`, `2026Q2`)와 함께 생성되고 ID가 반환된다.
2. **Given** 빈 목록, **When** `todo add "장보기"` (태그 없음), **Then** 새 항목이 태그 없이 생성된다 (기존 동작 보존).
3. **Given** 빈 목록, **When** 사용자가 6개 이상의 태그를 지정해 `todo add` 호출, **Then** 항목은 생성되지 않고 "태그는 최대 5개까지 허용됩니다"는 취지의 오류 메시지가 출력된다.
4. **Given** 빈 목록, **When** 사용자가 21자 이상인 태그를 포함해 `todo add` 호출, **Then** 항목은 생성되지 않고 "태그 길이는 20자 이내여야 합니다"는 취지의 오류 메시지가 출력된다.
5. **Given** 빈 목록, **When** 동일한 태그를 두 번 지정(`--tag work --tag work`), **Then** 항목은 정상 생성되며 태그는 중복 없이 1개로 저장된다.

### User Story 2 - 태그로 목록 필터링 (Priority: P1)

사용자는 목록 조회 시 특정 태그를 가진 항목만 빠르게 추려낼 수 있다.

**Why this priority**: 태그가 부여되어도 조회 시 활용 수단이 없으면 사용 가치가 거의 없다. US1과 함께 묶어야 MVP가 의미를 가지므로 P1.

**Independent Test**: US1로 서로 다른 태그 조합을 가진 항목들을 추가한 뒤 `todo list --tag <name>`을 실행하여 해당 태그를 가진 항목만 반환되는지 확인.

**Acceptance Scenarios**:

1. **Given** 태그가 다른 항목 다수가 존재함, **When** `todo list --tag work`, **Then** `work` 태그를 가진 항목만 표시된다.
2. **Given** 어떤 항목도 해당 태그를 갖지 않음, **When** `todo list --tag urgent`, **Then** "(항목이 없습니다)" 또는 동등한 빈 결과가 반환되며 오류는 발생하지 않는다.
3. **Given** 항목들이 추가되어 있음, **When** `todo list` (태그 옵션 없음), **Then** 기존 동작(전체 또는 다른 필터 결과)이 그대로 유지된다 (회귀 없음).
4. **Given** 항목들이 추가되어 있음, **When** `todo list --tag work --completed`, **Then** `work` 태그이면서 완료 상태인 항목만 표시된다 (다른 필터와 결합 가능).

---

### Edge Cases

- **태그 정규화**: 사용자가 대소문자나 앞뒤 공백을 다르게 입력해도(`Work`, `work `, ` work`) 동일 태그로 취급할지(소문자 정규화 + trim) 명세는 정규화를 적용하는 것을 기본 가정으로 한다(아래 Assumptions 참고).
- **빈/공백만의 태그**: 빈 문자열이나 공백만으로 구성된 태그(`--tag ""`, `--tag "   "`)는 항목이 생성되지 않고 오류 메시지를 반환한다.
- **태그에 사용 가능한 문자**: 영문/숫자/한글/하이픈/언더스코어(`-`, `_`)만 허용하며, 그 외 문자(공백 포함, 콤마 포함)는 오류로 거부한다 — 사용자가 태그 구분자와 혼동하지 않도록.
- **기존 데이터 호환**: 본 기능 도입 이전에 저장된 항목(태그 필드 없음)은 빈 태그 목록을 가진 것으로 해석되어야 하며, 추가 작업 없이 정상 조회·완료·삭제가 가능해야 한다.
- **여러 태그를 한 항목에 부여한 후 한 태그로 필터링**: 항목이 `work`와 `2026Q2`를 모두 가지면 `--tag work` 또는 `--tag 2026Q2` 둘 다에서 검색된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-101**: 시스템은 `todo add` 명령에서 `--tag <name>` 옵션을 0회 이상 받을 수 있어야 한다 (반복 지정으로 다중 태그 표현).
- **FR-102**: 시스템은 한 항목에 부여 가능한 태그 수를 **최대 5개**로 제한해야 한다. 한도 초과 시 항목을 생성하지 않고 오류 메시지를 반환해야 한다.
- **FR-103**: 시스템은 각 태그의 길이를 **1자 이상 20자 이하**로 제한해야 한다 (정규화 후 길이 기준). 위반 시 항목을 생성하지 않고 오류 메시지를 반환해야 한다.
- **FR-104**: 시스템은 태그 입력에 대해 정규화(앞뒤 공백 제거, 소문자 변환)를 적용한 뒤 동일 항목 내 중복을 제거해야 한다.
- **FR-105**: 시스템은 태그가 허용 문자 집합(영문 대소문자/숫자/한글/`-`/`_`)만 포함하도록 검증하고, 그 외 문자가 포함되면 오류 메시지를 반환해야 한다.
- **FR-106**: 시스템은 태그 옵션이 전혀 없는 `todo add` 호출에 대해 기존 동작과 동일하게 항목을 생성해야 한다 (기존 사용자 흐름 보존).
- **FR-107**: 시스템은 `todo list --tag <name>` 호출 시 해당 태그를 (정규화 후) 포함한 항목만 반환해야 한다.
- **FR-108**: `todo list --tag` 옵션은 기존 `--completed`/`--pending`/`--priority` 필터와 **AND 결합**으로 동시에 적용 가능해야 한다.
- **FR-109**: 본 기능 도입 이전에 생성되어 태그 필드가 없는 기존 저장 데이터를 시스템이 빈 태그 목록으로 해석하여 정상 조회·완료·삭제할 수 있어야 한다 (스키마 마이그레이션 무중단).
- **FR-110**: `todo list` 출력 형식에 각 항목의 태그가 표시되어야 하며 (예: `tags:[work,2026Q2]`), 태그가 없으면 비어 있음을 알아볼 수 있도록 표기되어야 한다 (예: `tags:[]` 또는 `-`).

### Key Entities

- **ToDoItem (확장)**: 기존 속성(`id`, `title`, `due_date`, `priority`, `completed`, `created_at`)에 더해 `tags`(0~5개의 정규화된 문자열 집합) 속성을 추가로 가진다. 태그 집합은 순서가 의미를 가지지 않으며 중복을 허용하지 않는다.
- **Tag (값 객체)**: 1~20자 길이의 정규화된(소문자, trim) 문자열로, 허용 문자 집합 제한을 따른다. 별도 엔티티로 영속되지는 않고 `ToDoItem` 내부 컬렉션 값으로만 존재한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-101**: 태그 1~5개를 지정한 `todo add` 명령이 95% 사례에서 1초 이내에 완료된다 (기존 SC-001 임계와 동일).
- **SC-102**: 100개 항목 중 특정 태그를 가진 항목 5개를 `todo list --tag <name>`으로 조회 시 결과 정확도(정밀도·재현율) 100%이며 95% 사례에서 1초 이내 반환된다.
- **SC-103**: 본 기능 배포 후 기존 사용자의 태그 미사용 흐름(`todo add "..."`, `todo list`)에 대한 회귀 0건 — 전체 기존 통합 테스트가 그대로 통과해야 한다.
- **SC-104**: 본 기능 도입 이전에 생성된 저장 파일(`todo.json`, 태그 필드 없음)을 추가 마이그레이션 단계 없이 새 버전으로 읽고 모든 기존 명령(추가/조회/완료/삭제)이 100% 정상 동작한다.

## Assumptions

- 태그는 **클라이언트 측 정규화**(앞뒤 공백 제거 + 소문자 변환)를 거쳐 저장된다. 사용자가 `Work`, `work`, ` WORK ` 중 무엇을 입력해도 동일 태그로 처리된다.
- 태그 허용 문자는 영문 대소문자/숫자/한글/하이픈/언더스코어이며, 공백·콤마·기타 특수문자는 거부한다. (콤마는 향후 콤마 구분 입력 형식과 충돌 방지를 위함.)
- 태그 검색은 정확 일치(정규화 후)이며 부분 일치·정규식 매칭은 본 명세 범위 밖이다.
- `todo list --tag` 옵션은 단일 태그명만 받는다(반복 지정 시 마지막 값 사용 — argparse 기본 동작). 다중 태그 AND/OR 조회는 본 명세 범위 밖이며 향후 별도 기능으로 검토.
- 기존 `todo.json`(태그 필드 없음)은 빈 태그 목록(`[]`)을 가진 것으로 해석한다. 명시적 마이그레이션 명령은 도입하지 않는다.
- 본 기능은 헌법 원칙 5(CLI 전용)를 그대로 따르며 표준 입출력만 사용한다.

## Constitution Constraints (프로젝트 헌법 관련 필수 제약)

- **인터페이스 범위**: 본 기능은 CLI(터미널) 기반으로 한정된다. REST API/GUI/웹 인터페이스는 범위 밖.
- **테스트 우선**: 모든 신규 시나리오에 대해 RED 테스트를 먼저 작성한 뒤 구현한다. 우선 작성될 테스트 목록(예시):
  - `tests/unit/test_tags.py` — 태그 정규화·검증 (개수/길이/문자 집합/중복)
  - `tests/unit/test_service.py` (확장) — `TodoService.add`/`list`의 태그 인자 처리 및 비존재 태그 필터
  - `tests/unit/test_storage_json.py` (확장) — 태그 직렬화/역직렬화, 기존 데이터(필드 없음) 후방 호환
  - `tests/integration/test_cli_add.py` (확장) — `--tag` 반복 옵션, 한도 초과·길이 초과 오류 경로
  - `tests/integration/test_cli_list.py` (확장) — `--tag` 단독·`--completed`/`--priority`와의 결합 필터
- **최소 의존성**: 본 기능은 표준 라이브러리만으로 구현 가능하므로 신규 외부 패키지를 추가하지 않는다.
- **레이어 분리 요구**: 태그 정규화·검증 규칙은 `services/` 계층(예: `services/tags.py` 또는 `services/models.py`)에 위치하고, `cli/`는 입력 수집과 메시지 출력만 담당한다. `storage/`는 태그 컬렉션의 직렬화/역직렬화만 책임진다.
